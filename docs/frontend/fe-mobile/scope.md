# fe-mobile — Scope

## Purpose

fe-mobile turns the Ionic/Angular web app (`apps/frontend-ng/`) into deployable iOS and
Android native apps via Capacitor. It delivers: a Capacitor project scaffold (native
platform directories, `capacitor.config.ts`), SecureStorage for the session token,
push notifications (FCM + APNs via `@capacitor/push-notifications`), status bar and
splash-screen polish, touch-UX hardening for the shell and any remaining web-only
interactions, and a reproducible signed-build CI pipeline (GitHub Actions). When this
module is done, a single git push to `main` can produce a signed `.ipa` and a signed
`.aab` ready for TestFlight / Google Play internal testing.

## Binding upstream decisions

- **Web → memory token, Mobile → SecureStorage** — the `TokenStorage` abstraction and
  `SecureTokenStorage`/`SecureStoragePort` interfaces are already scaffolded in
  `apps/frontend-ng/src/app/core/auth/token.storage.ts`. fe-mobile must provide the
  Capacitor adapter and swap the DI provider.
  → [fe-core/open-questions.md](../fe-core/open-questions.md) Q2

- **Socket auth via `auth.token` payload (not cookies)** — the Capacitor WebView does not
  reliably propagate httpOnly cookies; `SocketService` already uses bearer token in its
  handshake `auth` payload, so socket auth works natively with no fe-mobile changes.
  → [fe-core/open-questions.md](../fe-core/open-questions.md) Q5

- **Navigation: side rail ≥ 1080 px / overlay drawer below / bottom-nav bar on mobile** —
  all three shells are already implemented (`ShellComponent`, `RailComponent`,
  `BottomNavComponent`). No new navigation component is needed; fe-mobile only verifies
  the bottom-nav bar renders correctly in the Capacitor WebView safe-area context.
  → [fe-ui-kit/open-questions.md](../fe-ui-kit/open-questions.md) Q1

- **Angular: standalone components + signals, no NgModules** — applies to any service or
  component added in this module (e.g., `PushNotificationService`, `SplashService`).
  → [fe-foundation/open-questions.md](../fe-foundation/open-questions.md) Q1

- **Capacitor/core 8.4.0 already installed** — `@capacitor/core@8.4.0` is in
  `apps/frontend-ng/package.json` (pinned, as per fe-foundation Q7). The CLI
  (`@capacitor/cli`) and platform packages (`@capacitor/ios`, `@capacitor/android`) must
  be added to the same `apps/frontend-ng/package.json` at matching 8.x versions.
  → [fe-foundation/open-questions.md](../fe-foundation/open-questions.md) Q7

## Target structure

```
apps/frontend-ng/
├── capacitor.config.ts              # Capacitor project config (new)
├── ios/                             # Xcode project (generated by `cap add ios`)
├── android/                         # Gradle project (generated by `cap add android`)
└── src/app/
    ├── core/
    │   ├── auth/
    │   │   └── token.storage.ts     # Already has SecureTokenStorage (extend, not replace)
    │   └── native/                  # NEW — mobile-only services
    │       ├── secure-storage.adapter.ts   # Capacitor SecureStorage → SecureStoragePort
    │       ├── push-notification.service.ts
    │       ├── status-bar.service.ts
    │       └── splash.service.ts
    └── shell/
        └── shell.component.ts       # Extend: safe-area CSS var, SplashService on init
```

Generated native directories (`ios/`, `android/`) are committed to the repo so the CI
pipeline does not need `cap add`. Updated by `cap sync` on every build.

## Legacy inventory replaced

The legacy Next.js app (`apps/frontend/`) has no Capacitor or native mobile code. There
is no behavioral parity target for this module — parity means "the web app also works in
the Capacitor WebView without regressions."

| Legacy (apps/frontend/src) | Lines | Replacement |
|---|---|---|
| `public/manifest.json` | — | `capacitor.config.ts` + iOS/Android native manifests (not a direct port; the legacy manifest is a minimal PWA stub with `{"name":"iotpilot"}`) |
| No push notification code | n/a | `PushNotificationService` + FCM/APNs plugin |
| No SecureStorage | n/a | `SecureStorageAdapter` over `@capacitor-community/secure-storage` |
| No native build pipeline | n/a | GitHub Actions workflows for signed iOS + Android builds |

## Endpoints consumed

fe-mobile adds one new endpoint requirement:

| Endpoint | Used by | Status |
|---|---|---|
| `POST /users/me/push-token` | `PushNotificationService` — registers FCM/APNs token after permission grant | **Missing from spec** — see open-questions.md Q3 |
| `DELETE /users/me/push-token` | `PushNotificationService` — deregisters on logout | **Missing from spec** — see open-questions.md Q3 |
| `GET /settings/notifications` + `PUT /settings/notifications` | Already used in fe-settings; `pushNotifications` field controls the in-app toggle | Present ✅ |
| `GET /monitoring/alerts` | Alert push payload deeplinks to MonitoringPage (fe-dashboard) | Present ✅ |

All existing endpoints consumed by page modules work unchanged in the Capacitor WebView
(same `Authorization: Bearer` transport, same CORS configuration).

## Dependencies

All page modules must be done (fe-admin is the last, currently pending). fe-mobile does
not add new pages; it wraps the completed app into native targets. This module depends on
every prior module having passed `/fe-check` so the web build that `cap sync` copies into
the native projects is stable.

## Out of scope

- **New feature pages** — fe-mobile does not port any UI screens. All pages are done by fe-admin.
- **PWA / service worker** — no service worker strategy for the Angular app. Native push
  via Capacitor replaces the need for a PWA push solution.
- **Deep links / Universal Links in-app routing** — parsing a deeplink URL and routing
  to the correct Angular page is included (T6 safe-area + deeplink config), but full
  marketing-site app-site-association files for Universal Links are a post-launch DevOps
  task.
- **App Store / Play Store listing copy, screenshots** — store metadata is not code;
  this module only produces a pipeline that can upload a signed build to the store.
- **Xterm.js / SSH terminal** — handled by fe-device-advanced. No touch-specific
  changes needed beyond what the `SshTerminalComponent` already provides.
- **Fastlane** — the CI pipeline uses GitHub Actions + official `action-setup-xcode` /
  Gradle directly. Fastlane is an option but adds a Ruby toolchain; it is listed in
  open-questions.md Q5 for the team to decide.
- **fe-cutover** — Traefik switch and legacy removal are a separate module.
